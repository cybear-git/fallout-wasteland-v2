### 2. Обновление API поиска
**Файл:** `public/api/search.php`

#### Добавлена функция кэширования настроек:
```php
function getSetting($pdo, string $key, float $default = 0.0): float {
    static $cache = [];
    if (isset($cache[$key])) {
        return $cache[$key];
    }
    $stmt = $pdo->prepare("SELECT setting_value FROM game_settings WHERE setting_key = ?");
    $stmt->execute([$key]);
    $value = $stmt->fetchColumn();
    $result = $value !== false ? (float)$value : $default;
    $cache[$key] = $result;
    return $result;
}
```

#### Все шансы теперь читаются из БД:
- ✅ Базовый шанс лута (`$lootBaseChance`)
- ✅ Шанс крышек (`$capsChanceBase`, `$capsMin`, `$capsMax`)
- ✅ Шансы на типы предметов (`$weaponChance`, `$armorChance`, `$consumableChance`)
- ✅ Модификаторы характеристик (`$luckBonusMultiplier`, `$locationQualityMultiplier`)
- ✅ Pity timer и кулдаун (`$pityTimerThreshold`, `$cooldownSeconds`)
- ✅ Шансы на монстров (`$monsterEncounterBase`, `$monsterDangerMultiplier`)

## 🎮 Игровой процесс

### Как работает система:

1. **Игрок нажимает "Поиск"** → Запрос к `/api/search.php`
2. **Сервер читает настройки из БД** → Кэширует для производительности
3. **Рассчитывает модифицированные шансы**:
   ```
   armorChance = base_armor_chance + (luck * luckBonusMultiplier * 0.5) * locationMult
   ```
4. **Бросает кубик (1-1000)** → Определяет тип предмета
5. **Проверяет pity timer** → Если 50+ поисков без редкого, гарантирует находку
6. **Выдает результат** → Предмет, крышки или монстр

### Пример расчета:
```
Персонаж: Luck=7, Location Quality=2
Базовый шанс оружия: 0.4%

luckBonus = 7 * 0.3 = 2.1%
locationMult = 1 + (2 * 0.15) = 1.3
weaponThreshold = min(5, (0.4 + 2.1 * 0.8) * 1.3) = min(5, 2.7) = 2.7%

Итоговый шанс: ~2.7% (вместо базовых 0.4%)
```

## 🔒 Безопасность и производительность

### Принятые меры:
- ✅ **Кэширование настроек** → Один запрос на настройку за сессию
- ✅ **Prepared statements** → Защита от SQL-инъекций
- ✅ **Валидация типов** → `(int)`, `(float)` для всех значений
- ✅ **Fallback на дефолты** → Если настройки нет в БД, используется значение по умолчанию
- ✅ **Транзакции** → Атомарность операций с инвентарем

## 📊 Мониторинг и балансировка

### Логи для анализа:
Таблица `search_logs` содержит:
- `player_id` - кто искал
- `result` - что нашел (found_item, monster_encounter, nothing)
- `item_found_key` - ключ найденного предмета
- `xp_gained` - полученный опыт
- `created_at` - время поиска

### SQL-запросы для аналитики:
```sql
-- Средний шанс нахождения предметов за день
SELECT DATE(created_at) as date, result, COUNT(*) as count
FROM search_logs
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY DATE(created_at), result;

-- Топ игроков по количеству поисков
SELECT p.username, COUNT(*) as searches
FROM search_logs sl
JOIN players p ON p.id = sl.player_id
GROUP BY p.id ORDER BY searches DESC LIMIT 10;

-- Распределение найденных предметов
SELECT item_found_key, COUNT(*) as found_count
FROM search_logs
WHERE item_found_key IS NOT NULL
GROUP BY item_found_key ORDER BY found_count DESC;
```

## 📝 Заметки разработчика

### Почему выбран такой подход:
1. **Гибкость** → Админ может менять баланс без перезапуска сервера
2. **Прозрачность** → Все настройки видны в одном месте
3. **Безопасность** → Нет правок кода → нет риска внести баги
4. **Масштабируемость** → Легко добавить новые настройки

### Возможные проблемы и решения:
| Проблема | Решение |
|----------|---------|
| Частое изменение шансов ломает баланс | Ввести cooldown на изменение (логировать в admin_logs) |
| Игроки фармят в богатых локациях | Добавить diminishing returns для повторяющихся поисков |
| Pity timer слишком щедрый | Настроить порог через `search_pity_timer_threshold` |
